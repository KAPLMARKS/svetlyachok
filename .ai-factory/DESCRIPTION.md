# АИС «Светлячок»

## Обзор

Программный комплекс автоматизированной информационной системы (АИС) «Светлячок» — клиент-серверное решение для автоматизированного учёта посещаемости сотрудников (Time Tracking) на базе Wi-Fi-позиционирования внутри закрытых помещений.

Система использует метод сопоставления радиоотпечатков (Wi-Fi RSSI Fingerprinting) и алгоритмы машинного обучения (Weighted K-Nearest Neighbors, Random Forest) для классификации местоположения сотрудника (рабочее место, коридор, переговорная, вне офиса) без необходимости приобретения специализированного оборудования (BLE-маяки, RTT-роутеры, биометрические терминалы).

Проект разрабатывается в рамках магистерской диссертации по теме «Исследование методов Wi-Fi-позиционирования для систем учёта посещаемости в закрытых помещениях».

## Основные функции

### Мобильное приложение (сотрудник)

- Авторизация по корпоративным учётным данным (JWT)
- Фоновое сканирование Wi-Fi-сетей и сбор метрик RSSI от видимых точек доступа
- Передача радиоотпечатков на сервер по защищённому HTTPS
- Режим администратора для калибровки эталонной радиокарты помещений

### Веб-панель администратора

- Управление радиокартой (просмотр, добавление, редактирование точек калибровки)
- Управление сотрудниками и зонами (рабочие места, переговорные, коридоры)
- Отчёты по посещаемости, опозданиям, переработкам
- Визуализация точности позиционирования (метрики RMSE, вероятность обнаружения зоны)

### Серверная часть (Backend)

- REST API для мобильного клиента и веб-панели
- Приём и обработка радиоотпечатков по алгоритму WKNN и Random Forest
- Классификация местоположения на основе обученной радиокарты
- Автоматический расчёт отработанных часов с фиксацией опозданий и переработок
- REST API для интеграции с внешними ERP/HRM-системами (1C и аналоги)
- Метрологический контроль точности по стандарту ISO/IEC 18305:2016

## Ограничения, явно зафиксированные в методологии исследования

Эти ограничения являются методологическими и должны быть отражены в разделе «Ограничения» магистерской диссертации.

### Только Android для мобильного клиента

iOS не поддерживается в экспериментальной части исследования. Причина: API `NEHotspotHelper` (единственный способ получить полный список окружающих Wi-Fi-сетей с RSSI) требует специального entitlement от Apple, выдаваемого только операторам Wi-Fi-сетей и MDM-провайдерам. Это ограничение проверяется операционной системой iOS на этапе выполнения и не обходится через debug-сборку.

Это ограничение не снижает научную ценность исследования: алгоритмы WKNN и Random Forest, исследуемые в работе, платформо-независимы. Экспериментальная часть проводится на платформе Android, что является общепринятой практикой в академических исследованиях по Wi-Fi-позиционированию.

### Старая Wi-Fi инфраструктура (без FTM/RTT)

Вуз, в котором проводятся полевые испытания, использует стандартные Wi-Fi-точки доступа без поддержки протоколов измерения времени пролёта сигнала (IEEE 802.11mc / RTT). Это исключает методы time-of-flight позиционирования и определяет выбор RSSI Fingerprinting как единственного применимого метода в рамках существующей инфраструктуры.

### Без дополнительного аппаратного обеспечения

В исследовании не используются BLE-маяки, ультразвуковые передатчики и другое специализированное оборудование. Цель — подтвердить применимость стандартной корпоративной Wi-Fi-инфраструктуры для учёта посещаемости.

## Архитектура

Подробное описание архитектурных решений и зависимостей слоёв — в `.ai-factory/ARCHITECTURE.md`.

**Паттерн:** Clean Architecture (для backend), слоистая архитектура (для mobile), feature-based (для web).

## Архитектурные принципы

### Слои серверного приложения

- **API layer (FastAPI routers)** — приём HTTP-запросов, валидация Pydantic-схем, аутентификация
- **Service layer** — бизнес-логика (классификация, расчёт времени, создание отчётов)
- **ML layer** — обучение и применение моделей WKNN/Random Forest (отдельный модуль для воспроизводимости)
- **Repository layer** — работа с PostgreSQL через SQLAlchemy
- **Domain models** — Pydantic-модели и SQLAlchemy ORM-модели

### Архитектура мобильного приложения

- **UI layer** — Flutter widgets, Material Design
- **State management** — Riverpod или Bloc (выбор делается на этапе планирования)
- **Wi-Fi service** — обёртка над `wifi_scan`/`wifi_iot`, фоновое сканирование через WorkManager
- **API client** — HTTP-клиент (Dio) для общения с backend
- **Local cache** — локальное хранилище неотправленных радиоотпечатков (sqflite) на случай отсутствия связи

### Архитектура веб-панели

- **React SPA** + React Router
- **State management** — TanStack Query (server state) + Zustand (UI state)
- **API client** — общий с типами, сгенерированными из OpenAPI-схемы FastAPI

## Нефункциональные требования

- **Логирование:** настраиваемый уровень через переменную окружения `LOG_LEVEL`. Формат — структурированный JSON (`structlog`) для удобной обработки в production
- **Обработка ошибок:** структурированные ответы (`detail`, `code`, `correlation_id`) согласно RFC 7807 Problem Details
- **Безопасность:**
  - HTTPS обязательно для production
  - JWT с refresh token, ротация ключей
  - Rate limiting на endpoint авторизации
  - Хеширование паролей через bcrypt
  - Защита от CSRF для admin-панели (SameSite cookies)
- **Метрологический контроль:** автоматическое вычисление RMSE и вероятности обнаружения зоны (Detection Probability) согласно ISO/IEC 18305:2016 на каждом запуске тестового набора
- **Воспроизводимость экспериментов:** все параметры обучения моделей, версии библиотек, и набор калибровочных данных фиксируются в отдельных артефактах (для научных публикаций)
- **Интеграция с 1С/ERP:** REST API с OpenAPI-документацией для импорта данных посещаемости

## Этапы разработки

1. **Аналитика и проектирование** — сбор требований, ER-диаграмма БД, OpenAPI-спецификация
2. **UI/UX дизайн** — макеты мобильного приложения (Figma/Pencil) и веб-панели администратора
3. **Backend разработка** — реализация REST API, доменных моделей, ML-модуля WKNN/Random Forest
4. **Mobile разработка (Android)** — Flutter-приложение, модули сканирования и фоновой синхронизации
5. **Web разработка (admin)** — React SPA для управления радиокартой и отчётов
6. **Полевые испытания и калибровка** — измерения в реальных помещениях вуза, оптимизация радиокарты
7. **Метрологическая оценка** — расчёт RMSE и Detection Probability на тестовом наборе
8. **Развёртывание** — Docker-инфраструктура backend, Android APK для тестирования, веб-панель в production
