# Backend АИС «Светлячок»

Серверная часть системы автоматизированного учёта посещаемости на базе Wi-Fi RSSI fingerprinting.

## Стек

- Python 3.12+
- FastAPI (async REST API)
- SQLAlchemy 2.x + asyncpg + Alembic (PostgreSQL)
- pydantic-settings, structlog
- pytest, ruff, mypy

## Установка

```bash
cd backend
python -m venv .venv
source .venv/bin/activate     # Linux/macOS
.venv\Scripts\activate        # Windows
pip install -e ".[dev]"
```

## Конфигурация

Скопируйте `.env.example` в `.env` и заполните обязательные поля (`DATABASE_URL`, `JWT_SECRET`).

```bash
cp .env.example .env
```

`DATABASE_URL` должен использовать драйвер asyncpg, например:

```
DATABASE_URL=postgresql+asyncpg://svetlyachok:dev_password@localhost:5432/svetlyachok
```

## Локальный PostgreSQL

Самый быстрый способ — одноразовый docker-контейнер. Полноценный compose с volume и пользовательской настройкой добавим на вехе «Развёртывание».

```bash
docker run -d \
  --name svetlyachok-postgres \
  -e POSTGRES_USER=svetlyachok \
  -e POSTGRES_PASSWORD=dev_password \
  -e POSTGRES_DB=svetlyachok \
  -p 5432:5432 \
  postgres:16-alpine
```

Если Docker недоступен — поднимите Postgres любым удобным способом и убедитесь, что `DATABASE_URL` указывает на работающий инстанс.

## Миграции (Alembic)

Перед первым запуском применить все миграции:

```bash
alembic upgrade head
```

| Команда | Назначение |
|---------|-----------|
| `alembic upgrade head` | Применить все pending-миграции |
| `alembic downgrade -1` | Откатить последнюю миграцию |
| `alembic history -v` | История миграций |
| `alembic current` | Текущая ревизия БД |
| `alembic revision --autogenerate -m "описание"` | Сгенерировать новую миграцию по diff моделей и БД |

**Правило:** каждая autogenerated-миграция должна быть проверена вручную перед коммитом.
Особое внимание — partial-индексам (`postgresql_where`), enum-изменениям и `server_default`:
autogenerate их часто упускает или дополняет неточно.

## Seed-данные

Для быстрого наполнения dev-БД тестовым набором (зоны, тестовый сотрудник, калибровочные fingerprints):

```bash
python scripts/seed.py
```

Скрипт идемпотентен: повторный запуск ничего не сломает (использует `INSERT ... ON CONFLICT DO NOTHING` по уникальным колонкам).

## Тестирование с реальной БД

Интеграционные тесты (`pytest -m integration`) поднимают тестовый PostgreSQL по одной из стратегий:

1. Переменная `TEST_DATABASE_URL` указывает на готовую тестовую БД (рекомендуется в CI).
2. Иначе используется [testcontainers](https://testcontainers-python.readthedocs.io/) — поднимается контейнер `postgres:16-alpine` (требуется Docker).
3. Если ни Docker, ни `TEST_DATABASE_URL` не доступны — интеграционные тесты автоматически пропускаются (`pytest.skip`).

Каждый тест получает «чистую» БД через savepoint-rollback: миграции применяются один раз на pytest-сессию, изменения откатываются после каждого теста.

Дополнительно: тест `test_downgrade_then_upgrade_idempotent` запускается только при `RUN_MIGRATION_CYCLE_TEST=1` — он меняет схему и не дружит с savepoint-rollback.

| Метод | Путь | Назначение |
|-------|------|-----------|
| `GET` | `/api/v1/health` | Healthcheck (liveness + реальный `SELECT 1` к БД) |
| `POST` | `/api/v1/auth/login` | Вход по email + паролю → пара JWT (access + refresh) |
| `POST` | `/api/v1/auth/refresh` | Обмен refresh-токена на новый access |
| `GET` | `/api/v1/me` | Информация о текущем пользователе (требует Bearer access) |

Полный OpenAPI: http://localhost:8000/docs (Swagger UI), http://localhost:8000/redoc.

## Аутентификация

API защищается JSON Web Tokens. Схема: `email + password` → пара токенов
(`access` 30 минут, `refresh` 7 дней) → клиент передаёт access в заголовке
`Authorization: Bearer <token>`.

### Получить пару токенов

```bash
curl -X POST http://localhost:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@svetlyachok.local","password":"admin12345"}'
```

Ответ:

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer",
  "expires_in": 1800
}
```

### Использовать access-токен

```bash
curl http://localhost:8000/api/v1/me \
  -H "Authorization: Bearer ${ACCESS_TOKEN}"
```

### Обновить access без повторного login

```bash
curl -X POST http://localhost:8000/api/v1/auth/refresh \
  -H "Content-Type: application/json" \
  -d '{"refresh_token":"<refresh-from-login>"}'
```

Refresh переиспользуется до истечения срока (без token rotation на этой
вехе — добавим вместе с blacklist'ом jti, если в полевых испытаниях
обнаружим повторное использование украденных токенов).

### Rate limiting

`/auth/login` ограничен **5 запросами в минуту на IP**, `/auth/refresh` —
**10 в минуту**. Превышение → `429 Too Many Requests` + заголовок
`Retry-After`. Лимиты конфигурируются переменными окружения
`AUTH_LOGIN_RATE_LIMIT` и `AUTH_REFRESH_RATE_LIMIT` (формат slowapi:
`<count>/<period>`).

### Безопасность

- Хеширование паролей: **bcrypt с work factor 12** (рекомендация OWASP 2024)
- Алгоритм JWT: **HS256**, секрет из `JWT_SECRET` (минимум 32 символа,
  валидируется при старте)
- Защита от user enumeration: при missing/inactive user всё равно тратится
  bcrypt-цикл (timing-safe), все три причины fail возвращают единый
  `401 invalid_credentials`
- Refresh-токен НЕ принимается как access (поле `type` в payload)
- Access-токен НЕ принимается как refresh
- Rate limiting на /auth-эндпоинтах против брутфорса
- В response никогда не возвращается `hashed_password`
- В логах никогда не появляются пароли или сами токены — только
  `correlation_id` и `employee_id`

**Production-заметки:**

- НИКОГДА не используй seed-пароли (`admin12345`, `employee12345`) в production
- Ротация `JWT_SECRET` инвалидирует все активные токены
- Для горизонтального масштабирования (несколько uvicorn worker'ов) перейти
  на slowapi с Redis-storage — текущая in-memory реализация даёт ограничение
  per-process

## Архитектура

Clean Architecture: domain → application → infrastructure → presentation.
Подробное описание — в `../.ai-factory/ARCHITECTURE.md`.

### Структура схемы БД

| Таблица | Назначение |
|---------|-----------|
| `employees` | Учётные записи сотрудников (роль `admin`/`employee`, расписание) |
| `zones` | Зоны учёта (рабочее место, коридор, переговорная, вне офиса) |
| `fingerprints` | RSSI-радиоотпечатки: калибровочные (с привязкой к зоне) и live |
| `attendance_logs` | Сессии присутствия сотрудника в зоне (с расчётом длительности) |

Дополнительные инварианты схемы:

- `fingerprints.is_calibration = true` ⇒ `zone_id IS NOT NULL` (CHECK)
- `attendance_logs.ended_at > started_at` (CHECK)
- Partial-index `ix_attendance_logs_open_sessions WHERE ended_at IS NULL` для быстрого ответа на «кто сейчас на работе»
