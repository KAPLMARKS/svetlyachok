# Локальный запуск АИС «Светлячок» — пилотная инструкция

> **Аудитория:** дипломный руководитель / комиссия / коллеги, желающие проверить
> работоспособность системы на одном компьютере (Windows 11) + одном
> Android-телефоне в общей Wi-Fi сети. Без Docker, nginx, production-конфигов.

Документ описывает:
1. Что нужно установить (Часть 1).
2. Как запустить backend + БД + web-панель (Часть 2).
3. Как собрать и установить Android APK (Часть 3).
4. Как пользоваться системой — пошаговый юзер-флоу (Часть 4).
5. Как проверить корректность работы — E2E-сценарий с критериями приёмки (Часть 5).
6. Что делать при типичных проблемах (Часть 6).

| Компонент | Версия | Зачем | Как установить |

## Часть 2. Запуск backend + БД + web

### 2.1. Клонирование

```powershell
git clone https://github.com/KAPLMARKS/svetlyachok.git
cd svetlyachok
```

### 2.2. PostgreSQL: создание БД и пользователя

Откройте `psql -U postgres` (пароль задавался при установке) и выполните:

```sql
CREATE USER svetlyachok WITH PASSWORD 'dev_password';
CREATE DATABASE svetlyachok OWNER svetlyachok;
GRANT ALL PRIVILEGES ON DATABASE svetlyachok TO svetlyachok;
\q
```

Альтернатива: выполнить те же команды в **pgAdmin 4** (GUI ставится вместе
с PostgreSQL).

### 2.3. Backend: venv, зависимости, миграции, seed

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\Activate.ps1     # активация venv в PowerShell
pip install -e ".[dev]"          # установка зависимостей и dev-инструментов

# Конфигурация
Copy-Item .env.example .env
# Откройте .env и поправьте JWT_SECRET (любая строка ≥ 32 символов).
# Сгенерировать можно так:
python -c "import secrets; print(secrets.token_urlsafe(48))"
# Скопируйте вывод в JWT_SECRET= в .env.

# Миграции БД
alembic upgrade head

# Заполнение seed-данными (4 зоны + admin + employee + 3 калибровки)
python scripts/seed.py
```

После этого в БД появятся:
- **Admin:** `admin@svetlyachok.local` / `admin12345`
- **Employee:** `employee@svetlyachok.local` / `employee12345`
- 5 зон (рабочие места, коридор, переговорная, вне офиса)

### 2.4. Узнайте локальный IP компьютера

В PowerShell:

```powershell
ipconfig | findstr IPv4
```

Найдите тот IP, по которому компьютер виден в Wi-Fi сети
(обычно `192.168.0.x`, `192.168.1.x` или `10.0.0.x`).
**Запомните его** — он пригодится для mobile-сборки.

### 2.5. Backend: запуск dev-сервера

```powershell
# Из директории backend/, при активированной .venv
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

Флаг `--host 0.0.0.0` обязателен, иначе телефон не достучится с другого
устройства в сети. Откройте **новый** терминал — этот должен оставаться запущен.

Проверка:
- В браузере на компьютере: http://localhost:8000/docs — Swagger UI с эндпоинтами.
- С телефона (в той же Wi-Fi): http://`<IP>`:8000/docs — должен открыться тот же UI.
  Если не открывается — см. Часть 6 «Файрвол».

### 2.6. Web: установка зависимостей и запуск

В отдельном PowerShell:

```powershell
cd web
npm install                       # один раз при первой настройке
npm run dev                       # → http://localhost:5173
```

Откройте http://localhost:5173 — должен появиться экран входа «Светлячок».

### 2.7. Подготовка радиокарты через web-панель

1. Войдите как admin (`admin@svetlyachok.local` / `admin12345`).
2. Сайдбар → **«Зоны»** → проверьте, что 5 seed-зон видны. При желании
   добавьте свои (тип, цвет).
3. Сайдбар → **«Радиокарта»** → seed создаёт 3 калибровочные точки.
   **Для качественной классификации нужно минимум 3 точки на каждую
   рабочую зону** (workplace / corridor / meeting_room). Их соберём через
   mobile в admin-режиме (см. Часть 4.B).

## Часть 3. Mobile: сборка и установка APK

### 3.1. Подготовка зависимостей

```powershell
cd mobile
fvm flutter pub get
fvm flutter pub run build_runner build --delete-conflicting-outputs
```

Команда `build_runner` нужна один раз при первой настройке (генерирует
`*.freezed.dart` и `*.g.dart`).

### 3.2. Сборка release-APK с правильным BACKEND_URL

```powershell
# Замените 192.168.1.42 на свой IP из шага 2.4
fvm flutter build apk --release `
  --dart-define=BACKEND_URL=http://192.168.1.42:8000
```

APK будет в `mobile/build/app/outputs/flutter-apk/app-release.apk`.

### 3.3. Установка на телефон

#### Вариант А — через ADB (USB-кабель + USB-debugging)

1. На телефоне: **Настройки → О телефоне → Номер сборки** (тапнуть 7 раз)
   → разблокирует «Параметры разработчика».
2. **Параметры разработчика → Отладка по USB → Включить.**
3. Подключите телефон USB-кабелем, на запросе «Разрешить отладку?» — Да.
4. ```powershell
   adb devices                                  # должен видеть устройство
   adb install -r mobile/build/app/outputs/flutter-apk/app-release.apk
   ```

#### Вариант Б — через файлообменник

1. Скопируйте `app-release.apk` на телефон (Bluetooth / Telegram /
   Google Drive).
2. Откройте файл из проводника телефона.
3. Android попросит разрешить «Установка из неизвестных источников» — разрешите.

### 3.4. Первый запуск приложения

При первом запуске Android попросит **разрешения на местоположение** — это
нужно потому, что Wi-Fi сканирование на Android требует location permission
(независимо от того, используется ли GPS).

Разрешите:
- «Доступ к местоположению» — обязательно;
- На Android 10+ для фоновой работы — **«Разрешать всегда»** (через
  «Настройки → Приложения → Светлячок → Разрешения»).

## Часть 4. Юзер-флоу: что и как делать

### 4.A. Web — главные сценарии (admin)

#### Вход
1. Открыть http://localhost:5173.
2. `admin@svetlyachok.local` / `admin12345` → попадаем на дашборд `/attendance`.
3. На F5 сессия восстанавливается через refresh-token (хранится в
   `localStorage`).

#### Дашборд «Кто где сейчас»
- Таблица всех активных сотрудников: **ФИО / email / текущая зона / время начала / статус**.
- Сверху — счётчики: «На рабочих местах: N», «В коридоре: M», «На переговорной: K», «Не на месте: L».
- Обновляется автоматически каждые 30 сек.
- Клик по строке → переход на детальную страницу сотрудника.

#### Детальная страница сотрудника
URL: `/attendance/:employeeId`
- **PeriodPicker** наверху: «Сегодня / Неделя / Месяц / Период».
- 4 карточки сводки: «Отработано», «Опозданий», «Переработка», «Сессий».
- **График часов по дням** (recharts BarChart).
- Таблица сессий: дата / с / до / длительность / зона / статус.

#### CRUD сотрудников (`/employees`)
- **Создать:** «+ Добавить» → форма (email, ФИО, роль, начальный пароль,
  расписание `HH:MM`).
- **Редактировать:** иконка карандаша → имя / роль / расписание / чекбокс
  «Очистить расписание».
- **Сменить пароль:** иконка ключа → admin-reset (без `old_password`).
- **Деактивировать:** иконка щита → confirm → пользователь больше не
  залогинится. Самого себя нельзя — кнопка скрыта.
- **Поиск:** debounced 300мс по email/ФИО.
- **Пагинация:** 20 на страницу.

#### CRUD зон (`/zones`)
- **Создать:** название, тип (Рабочее место / Коридор / Переговорная /
  Вне офиса), описание, цвет (palette + ручной HEX).
- **Удалить:** при попытке удалить зону, на которую ссылается attendance,
  backend вернёт `409 zone_in_use` → toast «Нельзя удалить: на зону
  ссылаются записи учёта».

#### Радиокарта (`/radiomap`)
- SVG-схема офиса (заглушка `office-floorplan.svg` 1200×800) с
  прозрачными прямоугольниками-зонами.
- Точки калибровки нарисованы кружочками внутри своих зон с
  детерминированным jitter'ом (5×5 сетка).
- **Hover** на точке → tooltip с `captured_at`, sample_count, BSSID count.
- **Click** на точку → confirm-модал «Удалить точку» → `DELETE
  /calibration/points/:id`.
- Сводка наверху: «X / Y зон откалибровано (≥ 3 точки)» — указатель
  готовности к ML-классификации.

#### Logout
- Topbar справа → **«Выйти»** → backend `POST /auth/logout` (best-effort) +
  локальная очистка → редирект на `/login`.

### 4.B. Mobile — флоу сотрудника и admin-калибровка

#### Сотрудник: «отметить присутствие»

1. **Splash** → автоматически: на экране короткий спиннер 6 секунд
   максимум.
2. **Login** → ввести email + пароль → автопереход на главный экран.
3. **Главный экран «Я на работе»**:
   - Карточка сверху с ФИО + ролью.
   - Большая кнопка **«Снять отпечаток сейчас»** — тап запускает Wi-Fi
     скан (учитывает Android-throttling: ≤ 4 сканов за 2 минуты).
   - Карточка статуса синхронизации: онлайн/офлайн + счётчик «В очереди:
     N» + кнопка **«Синхронизировать»** при `pending > 0`.
   - В правом верхнем углу AppBar — индикатор-точка:
     - **зелёная** = online + pending=0 (всё доехало);
     - **жёлтая** = pending > 0 (есть несинхронизированные);
     - **серая** = offline.
4. После тапа «Снять отпечаток» приложение:
   - снимает Wi-Fi скан (нормализует BSSID, дедуплицирует),
   - кладёт fingerprint в локальный sqflite-кэш с `sync_status=pending`,
   - сразу пытается отправить через `POST /api/v1/fingerprints/batch`
     (если онлайн).
5. **Фоновая работа:** WorkManager раз в 15 минут (минимум, который
   разрешает Android) запускает `scanAndCache` и `syncFingerprints`.
   Doze/App Standby могут отложить — это нормально, не real-time.

#### Сотрудник: settings + logout

- Drawer → **«Настройки»** → видны:
  - User info,
  - Кнопка ручной синхронизации,
  - Текущий BACKEND_URL (для отладки),
  - Версия приложения,
  - **«Выйти»** → backend `POST /auth/logout` + локальная очистка кэша
    → возврат на `/login`.

#### Admin: калибровка зон

(только для пользователя с `role=admin`)

1. Drawer → **«Калибровка зон»** (видна только админам).
2. Список зон из backend (`GET /zones`).
3. Тап по зоне → экран **«Капчер калибровки для <Зоны>»**.
4. Большая кнопка **«Снять эталонную точку»** — тап:
   - снимает Wi-Fi,
   - сразу шлёт `POST /api/v1/calibration/points` с `zone_id`,
   - на success — счётчик «Снято: N» инкрементируется (рекомендуется ≥ 3 на зону),
   - на success backend инвалидирует ML-кэш (классификатор переобучится при
     следующем запросе позиционирования).
5. Можно нажимать кнопку много раз подряд, не выходя с экрана — для набора
   3-5 точек подряд.
6. После 3+ точек закрыть экран → mobile сотрудника при следующем сканировании
   будет правильно классифицирован в эту зону.

## Часть 5. E2E-сценарий приёмки

> **Главный критерий «работающего MVP»:** mobile сотрудника отправляет
> fingerprint → backend классифицирует его в зону → web-панель показывает
> сотрудника в этой зоне с актуальным `work_hours`.

### Подготовка

- Backend на `:8000`, web на `:5173` (Часть 2 пройдена).
- APK установлен на телефон, разрешения на местоположение даны (Часть 3.4).
- Телефон и компьютер в одной Wi-Fi сети.
- Файрвол Windows пропускает входящие на 8000 (см. 6.1).

### Шаг 1. Калибровка через mobile (admin)

1. На телефоне войти как `admin@svetlyachok.local` / `admin12345`.
2. Drawer → «Калибровка зон» → выбрать зону **«Рабочее место А1»**.
3. Снять **3 эталонные точки** (3 тапа подряд по «Снять эталонную точку»
   с интервалом ~30 секунд между тапами — Android-throttling).
4. Повторить для **«Рабочее место Б1»** и **«Переговорная»**.
   Итого: ≥ 9 калибровочных точек в БД.

**Проверка через web (от admin):**
- Открыть http://localhost:5173/radiomap.
- Сводка должна показать **«3 / 3 зон откалибровано»** (для рабочего
  места + коридора + переговорной — рассмотрим только рабочие зоны).
- На SVG должны появиться кружочки в соответствующих прямоугольниках.

### Шаг 2. Сотрудник «приходит на рабочее место»

1. На телефоне выйти из admin-аккаунта (Settings → Выйти).
2. Войти как `employee@svetlyachok.local` / `employee12345`.
3. Подойти к зоне **«Рабочее место А1»** (физически, чтобы окружение
   Wi-Fi совпадало с тем, в котором калибровали).
4. Тап **«Снять отпечаток сейчас»**.
5. SnackBar «Отпечаток сохранён». Индикатор синхронизации зелёный, pending=0.

**Что произошло на backend:**
- `POST /api/v1/fingerprints/batch` с одним item.
- ML-классификатор (WKNN) определил зону по RSSI-вектору.
- Создалась/расширилась запись в `attendance_logs` с `zone_id` =
  «Рабочее место А1».

### Шаг 3. Web показывает «кто где»

1. На компьютере открыть http://localhost:5173/attendance (если уже
   открыто — просто подождать, дашборд обновляется каждые 30 сек, либо F5).
2. В таблице **employee@svetlyachok.local** должен иметь:
   - **Текущая зона:** «Рабочее место А1» (с цветной точкой).
   - **С:** время первого скана (HH:MM).
   - **Статус:** «Присутствует».
3. В счётчиках вверху: **«На рабочих местах: 1»**.
4. Клик по строке → детальная страница сотрудника:
   - Сводка «Отработано» > 0 секунд.
   - В таблице сессий — открытая сессия (без `до`).

### Шаг 4. Сотрудник «уходит на переговорную»

1. Подойти физически к зоне **«Переговорная»**.
2. Тап «Снять отпечаток сейчас».

**На backend:** новый fingerprint → классификация → закрылась предыдущая
сессия (с зоной «Рабочее место А1»), открылась новая (с зоной
«Переговорная»).

**На web:**
- В таблице у employee: **зона = «Переговорная»**.
- На детальной странице — две сессии: первая закрытая с длительностью,
  вторая открытая.

### Шаг 5. Проверка off-line + автоsync

1. На телефоне отключить Wi-Fi (или включить «Авиарежим»).
2. Тап «Снять отпечаток сейчас» — fingerprint сохраняется локально,
   pending=1 (индикатор жёлтый).
3. Включить Wi-Fi обратно. Через несколько секунд индикатор должен
   стать зелёным, pending=0 — сработал connectivity-listener (offline→online
   → авто-вызов `syncPending`).

### Шаг 6. F5 в web сохраняет сессию

1. На странице `/attendance` нажать F5.
2. Не должно бросить на `/login` — refresh-token из `localStorage`
   меняется на новый access через `POST /auth/refresh`, и сессия
   восстанавливается.

### Шаг 7. Logout в web и mobile

1. Web: Topbar → «Выйти» → редирект на `/login`. F5 → `/login` (токены
   стёрты).
2. Mobile: Settings → «Выйти» → возврат на login screen.

### Критерии приёмки

| # | Критерий | Где проверить |
|---|----------|---------------|
| 1 | Калибровка ≥ 3 точек на зону отображается на радиокарте | web `/radiomap` |
| 2 | Mobile-логин работает (admin и employee) | mobile login screen |
| 3 | Снятие fingerprint создаёт запись в backend | web `/attendance`, dev tools network |
| 4 | Backend правильно классифицирует зону по RSSI | web `/attendance` показывает зону, совпадающую с калибровкой |
| 5 | Перемещение сотрудника закрывает старую и открывает новую сессию | web детальная страница employee, видны 2 сессии |
| 6 | Offline-режим кэширует fingerprint локально | mobile индикатор → жёлтый, pending=1 |
| 7 | При появлении сети идёт автоsync | mobile индикатор → зелёный, pending=0 |
| 8 | F5 в web не выкидывает на login | web любая страница + F5 |
| 9 | `work_hours` детальной страницы > 0 | web `/attendance/:id` карточка «Отработано» |
| 10 | Logout стирает токены и редиректит | web/mobile после Logout F5 → login screen |

**Прохождение всех 10 пунктов = рабочий MVP.**

## Часть 6. Troubleshooting

### 6.1. Файрвол Windows блокирует подключение с телефона

**Симптом:** `http://<IP>:8000/docs` открывается на компьютере, но не
на телефоне.

**Решение:** разрешить входящие на порту 8000 в Windows Defender.

В PowerShell **с правами администратора**:

```powershell
New-NetFirewallRule -DisplayName "Svetlyachok backend (port 8000)" `
  -Direction Inbound -Action Allow -Protocol TCP -LocalPort 8000
```

Удалить правило позже:

```powershell
Remove-NetFirewallRule -DisplayName "Svetlyachok backend (port 8000)"
```

### 6.2. Mobile получает 404 / `Network unreachable`

- Проверить, что `BACKEND_URL` в команде сборки совпадает с актуальным IP
  компьютера (`ipconfig`).
- Проверить, что компьютер и телефон в **одной** Wi-Fi сети (некоторые
  гостевые Wi-Fi изолируют клиентов друг от друга — режим «AP isolation»).
- Перезапустить uvicorn с `--host 0.0.0.0` (без этого слушает только
  `localhost`).

### 6.3. CORS-ошибка в браузерной консоли web-панели

В dev-режиме Vite проксирует `/api → http://localhost:8000` (см.
`web/vite.config.ts`), поэтому CORS не должен возникать. Если возникает —
проверить, что не правили `web/src/lib/env.ts` и `VITE_API_BASE_URL=/api`
в `web/.env` (или вообще без `.env` — берётся default).

### 6.4. Mobile: «Сканер не нашёл точек доступа»

- Проверить permissions: Настройки → Приложения → Светлячок → Разрешения
  → Местоположение = «Разрешено».
- На Android 10+ для фона включить «Разрешать всегда».
- Включить службы геолокации (даже если GPS не нужен — Android требует
  это для Wi-Fi сканирования).
- Эмуляторы Android не показывают реальный Wi-Fi — тестировать только на
  физическом устройстве.

### 6.5. Мало калибровочных точек → классификатор отдаёт 503

Backend в эндпоинтах позиционирования возвращает `503` с code
`insufficient_calibration_points` если калибровка слишком разреженная.
Решение: набрать минимум **3 точки на каждую зону** через mobile в
admin-режиме (см. Часть 4.B).

### 6.6. Backend падает на старте: `database_url` / `jwt_secret` missing

Не скопирован `.env` файл или удалены поля. Скопируйте `.env.example` →
`.env`, заполните `JWT_SECRET` (см. 2.3). Перезапустите uvicorn.

### 6.7. `alembic upgrade head` падает с `connection refused`

PostgreSQL не запущен. На Windows: `services.msc` → найти
«postgresql-x64-16» → запустить.

### 6.8. Mobile: F5 / restart выкидывает на login

Проверить, что **secure storage** (refresh-token) не очищается. Если
проблема стабильно воспроизводится — переустановить APK (refresh-token
живёт 7 дней по умолчанию).

### 6.9. Web build падает на `npm run gen:api` (Backend not running)

`npm run gen:api` тянет `/openapi.json` с `:8000`. Сначала поднять
backend (Часть 2.5), потом запускать. Если backend не нужен — пропустить
этот шаг (web использует ручные DTO в `src/api/types.ts`).

### 6.10. Time / часовой пояс

Backend сериализует все datetime в UTC. Web и mobile отображают в
локальном TZ через `Intl.DateTimeFormat`. Если время выглядит «отстающим
на 3 часа» — это UTC, без перевода. На детальной странице сотрудника
видно ISO-время сессий (обновим под TZ-аware форматирование при
необходимости).
